By capturing rich semantic relationships between terms, LSR bridges the gap between lexical and neural retrieval, rivaling the effectiveness of dense and multivector encoders. Moreover, thanks to tailored inverted indexes for learned sparse representations~, LSR embeddings can be retrieved as efficiently as dense embeddings served by advanced data structures such as HNSW~. The main gap between LSR methods and their dense or multivector counterparts lies in training efficiency. A key bottleneck is the Language Model head, a module that projects the hidden states produced by the transformer backbone into a sparse vector representation anchored in a lexical vocabulary. In state-of-the-art LSR models such as ~, this component introduces substantial computational and memory overhead. As a first step, considering a batch of $B$ input sequences of size $S$ on a token vocabulary $$, the LM head computes and materializes the logits matrix $L ^{B S ||}$. Then, two element-wise operations---ReLU and ---followed by max-pooling over the sequence dimension, extract the salience scores of the most relevant terms, yielding a sparse output vector of $||$ dimensions. The bottleneck arises due to how this pipeline is implemented in standard frameworks such as PyTorch~. These operations execute sequentially: the logit matrix is computed block-wise in fast on-chip memory via an optimized matmul kernel, but must be written back to slower global memory due to its large size. It is then repeatedly transferred between SRAM and HBM for each subsequent operation . Moving data back and forth between SRAM and HBM is a highly suboptimal approach that fails to account for the memory-bound nature of the problem. First, materializing the full logit matrix in HBM creates an unwanted memory peak; on a BERT-like architecture, considering $B=S=512$ and $|| 30k$, the logit matrix requires $16$ GB to be stored. Observe that while modern GPUs can have up to 80 Gbytes of HBM, the SRAM is in the order of hundreds of kilobytes. This limits the maximum batch size, which is known to positively correlate with training speed and the effectiveness of the model~. In terms of speed, repeatedly reading this large logit matrix from HBM for processing in SMs and writing intermediate results back introduces significant delays due to data movement. This limits the exploitation of massive arithmetic throughput enabled by the thousands of parallel cores available on A100 or H100~. In this paper, we introduce , a fast---memory-efficient---kernel implemented in Triton~ tailored for the LM head in LSR models. Specifically, fuses all operations performed by the LM head, i.e., matmul, max, ReLU, and , on the logit matrix into a single Triton kernel . During the forward pass, we perform on-chip, online max-reduction, fused with ReLU and , so that we write only the max values and their indices to HBM. This eliminates the need to store the full intermediate logit matrix, dramatically reducing both memory usage and data movement overhead. For the backward pass, we reuse the stored max values and their indices to compute gradients for the hidden states and vocabulary embeddings. By doing so, we require only the hidden states at the max indices to be loaded and differentiated, saving additional compute and memory bandwidth. Our experiments demonstrate that our kernel, in isolation, achieves up to a $4.8 $ speedup and an order-of-magnitude reduction in peak memory usage compared to a PyTorch baseline. Each SM acts as a parallel processing engine powered by specialized computation units, such as INT32 for addressing, FP32 for standard arithmetic, or Tensor Cores for highly optimized matrix multiplication, managed by Warp Schedulers that select active warps to be dispatched and executed. The GPU memory hierarchy relies on High Bandwidth Memory and on-chip memory , which differ significantly in size and speed. The on-chip memory, located on SMs, is significantly faster than HBM but orders of magnitude smaller. For example, the NVIDIA H100 SXM5 features 80-94 GB of HBM with a bandwidth of 3 TB/s, yet only 256 KB of L1/Shared memory per each of its 132 Streaming Multiprocessors , with an estimated bandwidth of 30.9 TB/s~. To optimize data access latency, GPU kernels typically load data blocks from HBM to Shared Memory, perform computations, and write results back. Consequently, an efficient kernel must minimize data movement to/from HBM and maximize data reuse within on-chip memory. Operations executed on GPUs are classified as either or based on the balance between arithmetic throughput and memory access. In operations , execution time is dominated by mathematical computation; performance relies on maximizing the utilization of execution units. In contrast, operations spend significant cycles stalled waiting for data. For these kernels, the primary goal is to minimize global memory transfer and optimize access patterns to utilize the limited bandwidth effectively.
