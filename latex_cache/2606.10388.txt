It is that the retriever finds a relevant capability family and then exposes the wrong member of that family. We model each query as a choice among skills in a larger library, where two skills can share task vocabulary and procedure shape but differ in their execution consequence. Formally, a benchmark collection is $ =\{ , _i, _i,g^ )\}_{i=1}^{n}$, where $ _i$ is a user query, $ $ is its candidate pool, $ _i$ is the admitted helpful skill, $ _i$ is the query-specific execution-risk sibling, and $g^ : $ is the released benchmark family relation. The helpful skill guides execution toward a verifier-consistent solution for the current query. The risk sibling belongs to the same capability family but encodes a stale resource, missing precondition, wrong check, or otherwise query-inappropriate procedure. The labels are therefore query-specific: a skill can be useful in one context and a risky sibling in another. The benchmark construction keeps this distinction close to the decision a retriever must make. Starting from a task-facing skill that is admitted for a query, the paired sibling changes the condition that makes a procedure usable: the referenced resource, temporal scope, input precondition, formula-preserving operation, unit convention, or output schema. The sibling remains plausible under ordinary semantic retrieval, but following it would move the agent away from the query's verifier-facing target. The risk label is therefore an execution relation between a query and a skill, not an intrinsic safety label attached to the skill text. Candidate pools add library pressure around each pair. The helpful and risky siblings are ranked together with unrelated and partially related public skills, so the retrieval system must solve two coupled decisions: recover the active capability families from a larger library and choose the right representative within each active family. A retrieval system outputs a ranked list $R_K $ of at most $K$ skills. Recall@K and NDCG@K measure whether the helpful skill is present and ranked highly. Harmful sibling rate measures whether the query-specific risky sibling is exposed in the final top-$K$ list: \[ = {n} _{i=1}^{n} \] A relevance-only scorer can improve Recall@K and still increase HSR@K if it ranks both siblings highly. The target is therefore a utility-aware top-$K$ ranking: keep $ _i$ high while preventing $ _i$ from being exposed to the agent. SkillResolve factors same-capability skill selection into three components. A determines which candidates should compete as alternative representatives. A estimates how useful each candidate is for the current query. A keeps one highest-utility member from each resolved group before the final top-$K$ ranking. This factorization separates the benchmark relation from the deployable method interface. The released benchmark relation $g^ $ gives a controlled family source for measuring the within-family decision. Other family sources can be derived from public metadata, titles, or skill text. SkillResolve uses the same scoring and selection interface once a family source has produced candidate groups. Let $ $ denote a capability resolver. Given a query and candidate pool, it returns a set of active candidate groups \[ _{ }= ) = \{G_1 , ,G_m \} \] where each $G_j $ contains candidates that should be treated as alternative representatives for the same resolved capability. The groups cover the candidate pool, with singleton groups allowed. A singleton resolver recovers ordinary individual ranking; the released benchmark family relation gives a controlled grouping; public metadata and text-derived resolvers give public family sources. Capability resolution does not decide which skill is helpful. It defines the local competitions that the scorer must settle. If a helpful skill and a risky sibling are placed in the same resolved group, the representative selector can expose only the one assigned higher query-conditioned utility. If they are split into different groups, both can survive into the final ranking. The quality of $ $ therefore affects the recall--exposure tradeoff, while the utility scorer still decides which representative wins inside each group. SkillResolve learns a query-conditioned utility scorer $ $. For each training query $ _i$, the positive example is the admitted helpful skill $ _i$. The negative set $ _i$ contains confusable library alternatives mined under the same query and candidate-pool protocol, with $ _i \{ _i, _i\}$. These negatives are not random nonmatches and are not treated as universally unsafe skills. They are query-conditioned competitors: skills that appear plausible in the library but were not admitted as the helpful skill for $ _i$. The mining rule is fold-local. For each training query, SkillResolve ranks the fixed candidate pool with a base attribution-vote library signal, removes the admitted helpful skill and its paired risky sibling from the top 50, and keeps up to five highest-ranked remaining candidates as $ _i$. Dev and test queries never contribute training edges. The released benchmark relation is not used to mine negatives; it only defines representative competitions for controlled evaluation. The contract profile $ $ is a deterministic text profile over six execution-facing fields: resource binding, precondition, API or temporal scope, output schema, procedure, and identifiers. Keyword and literal detectors extract file-like resources, quoted identifiers, versions, schema terms, applicability terms, and procedural verbs from query or skill text. The difference is the execution-controlling clause: which resource to bind, which precondition to preserve, or which procedural contract to follow. These cases are documentation examples, not additional evaluation rows beyond the fixed 661-pair benchmark. One task asks an agent to estimate heart rate and breathing rate from 24 GHz continuous-wave radar recordings. Both siblings are radar-signal-processing skills: they describe I/Q parsing, clutter removal, range-bin or phase-signal extraction, and downstream vital-sign analysis. The helpful sibling ties these steps to the task-specific recording metadata and sidecar format. The risky sibling keeps the same radar pipeline but allows default artifact selection when several files or references are visible. The resulting failure is not an off-topic method; it is a plausible radar analysis run on the wrong capture or under the wrong I/Q convention, producing rates that can look numeric while no longer answer the requested instance. Another task asks the agent to fill a Word offer-letter template from employee data. Both siblings are document-generation skills for Word documents: they cover split template fields, headers and footers, nested tables, and template-preserving edits. The helpful sibling binds those operations to the requested template and the intended employee record. The risky sibling remains a relevant document skill, but weakens the binding to the task-specific file and data source. The execution consequence is a valid-looking document that may be generated from the wrong template, row, or field convention. A relevance ranker can see the right document-manipulation vocabulary in both siblings, while the executor would experience different document semantics. A code-generation task asks for a self-contained function that builds a folium map, computes pairwise geodesic distances, and raises ValueError when the input dictionary is empty. Both siblings are geopy skills and describe the same core operation: extract latitude and longitude fields, pass coordinate tuples to geodesic, and return a map with distance results. The helpful sibling treats the input contract as part of the procedure. The risky sibling keeps the same library guidance but relaxes the applicability condition when it conflicts with the query. The generated code can therefore import the right packages and follow the right distance formula while failing the oracle by skipping the required empty-input exception or by handling the dictionary schema incorrectly. SkillResolve-Bench 1.0 turns the task in Section~3 into a fixed-library retrieval problem. Each labeled instance provides a public query, a helpful skill, and a same-capability risky sibling whose resource binding, precondition, procedure, API scope, or example would steer execution away from the query's verifier-facing target. The 661 labeled pairs contain 630 answer-backed SRA-Bench pairs and 31 SkillsBench pairs with executable-test or oracle/verifier evidence. It fixes a query-disjoint 446/68/147 release split with no cross-split query leakage, no helpful/risky split mismatch, and no detected label cues after cue-cleaning and manual inspection. Main results use grouped five-fold held-out aggregation over all 661 pairs: each fold holds out one query-group partition for testing, uses the next for model selection, and trains learned scorers on the remaining partitions. Skill libraries change what retrieval must guarantee for language-model agents. Modern platforms treat skills as loadable operational artifacts with instructions, scripts, resources, and metadata . Recent work has scaled this idea into routing, retrieval, benchmarking, workflow orchestration, graph-structured skill representations, and governance systems . The retrieval layer therefore decides which procedural context enters the agent before planning or execution. This selection problem is sharper than ordinary relevance retrieval. Curated skills can improve execution, but gains weaken under realistic retrieval from large public collections, and public skill audits find weak routing metadata, non-actionable bodies, reusable-artifact defects, and technical debt . Recent skill-routing work further shows that top-$K$ skill quality cannot be reduced to independent query--skill relevance because the retrieved skills must also be compatible as a query-conditioned set . Security and governance studies add a second reason to treat skill retrieval as a trusted-use layer: loaded skill packages can affect planning, context, permissions, scripts, and local resources . The failure studied here appears when a library contains multiple skills from the same capability family. One skill may contain the current resource, precondition, or procedure needed by the task; a sibling may use the same domain vocabulary and procedure shape but point to a stale resource, skip a precondition, or apply the wrong check. A generic retriever can solve broad capability matching and still surface the wrong representative. The risky skill is not an irrelevant distractor; it is a close semantic neighbor with a different execution consequence. Structured retrieval and skill-set compatibility work reach compatible conclusions from other directions: relevant bundles may still leave entry points, roles, checks, or avoidances implicit, and compatible top-$K$ sets require supervision beyond independent relevance . This representative ambiguity is visible in public skill libraries. In a SkillRet audit using only public metadata, 4,716 of 4,997 queries have at least one non-gold skill in the gold skill's domain/action/object family; across seven standard retrievers, a top-three list surfaces such a sibling for 47.3\ of queries, and 36.6\ sibling. These siblings are not harmful labels. They show that, after a router finds the right capability family, it may still need to form the right representative competition. SkillResolve-Bench adds the execution-risk labels needed to evaluate that choice. {figures/skillresolve_concept.png} We formalize this setting as same-capability execution-risk skill retrieval. For each query, the benchmark identifies a helpful skill and a query-specific execution-risk sibling, then evaluates retrieval under a larger public skill pool. The central metric is harmful sibling rate : whether the risky sibling appears in the final top-K. A good skill retriever should have high Recall@K and NDCG@K for the helpful skill while keeping HSR@K low. This evaluation differs from positive skill retrieval benchmarks such as SkillRet and SRA-Bench, whose native labels identify useful or gold skills but do not assign query-specific execution-risk siblings. It also differs from malicious skill detection and permission enforcement: SkillResolve measures the upstream retrieval-time decision of which same-capability representative is admitted to the context. We propose SkillResolve, a utility-aware retrieval method for this setting. Figure~ summarizes the core decision made before execution. SkillResolve first resolves which candidates should compete as alternative representatives of an active capability, then scores each candidate with a query-conditioned utility model trained from admitted helpful skills and confusable library negatives. The utility model keeps ordinary retrieval signals and contract-profile cues for resource bindings, preconditions, API or temporal scope, output schema, and procedure indicators. The final selector keeps the highest-utility representative from each resolved family and ranks these representatives for the top-K list. The retrieval objective becomes a family-resolved utility competition before global top-K ranking. SkillResolve-Bench 1.0 makes this failure measurable as a reusable fixed-library retrieval task. It contains 661 helpful/risky sibling pairs, records source role and admission evidence, separates harmful-sibling labels from SkillRet library pressure, and releases hashes, query-disjoint splits, a fixed 7,982-candidate pool, and held-out outputs. Generic lexical, SkillRouter, and BGE reranking baselines retrieve relevant skills but expose risky siblings; Attribution-listwise suppresses them but loses helpful retrieval quality. SkillResolve reaches Recall@3 0.766, HSR@3=0, and NDCG@3 0.699, improving helpful retrieval over Attribution-listwise while both have HSR@3=0, and improving both retrieval and HSR over SkillRouter and BGE reranking.
