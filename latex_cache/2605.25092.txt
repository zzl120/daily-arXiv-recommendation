This is queried by the agent itself on almost every step, and the cost of those queries sits directly inside the agent's reasoning loop~. A representative load looks like this. A customer-support agent on its 800th turn of a three-month engagement holds roughly 5\,M records of memory. On each step it issues several lookups against that memory: the last similar issue on this account, the last successful fix for the same error code across account, the user's name and stated preferences from session~1. Each of these is a retrieval call that lives the agent's reasoning loop. Production agent traces show 5--20 such retrievals per agent step; if any one of them blocks for 50\,ms the user feels the agent stall. The retrieval system underneath the agent must therefore deliver . The IR systems agents currently inherit---Lucene, FAISS, PISA---were not built for this regime. Lucene-class engines treat the corpus as static, ignore the recency skew, and serve queries at 2--10\,ms latency. FAISS-class engines retrieve by dense similarity but lack BM25 and agent-specific metadata fusion. PISA's BlockMax-WAND degenerates on the long, multi-clause queries that agents emit when they ask their own memory natural questions. Even at parity quality, none of these systems is positioned to exploit what makes agent memory from general-purpose IR: Agent retrievals concentrate in the most recent fraction of records---far more sharply than web search or document retrieval. On LongMemEval, the median normalized rank of gold sessions for temporal/knowledge/multi-session questions is 0.20--0.27 ; a stress parameterization of 80/20 is comparable in steepness. Each record carries metadata: role (user, assistant, tool call/output, system, planning), session~ID, agent~ID, timestamps, and tool type. Agent memory contains natural language, code, API responses, and structured tool outputs---requiring both lexical and semantic matching. Agent memory grows monotonically, demanding sub-linear retrieval scaling to maintain interactive latency. The same agent fires factual lookups , reasoning queries , and temporal queries . No single fusion strategy wins on all three. These properties motivate a architecture fusing sparse lexical search with dense approximate nearest neighbor search~, augmented with agent-specific metadata filtering, temporal awareness, and a runtime selector that picks the fusion per query type rather than committing to one fixed strategy at deploy time. It is tempting to view agent memory as ``small IR'': dump the agent's traces into Elasticsearch or a vector database and serve queries with whatever ranker the team already maintains. This is how MemGPT~, Mem0~, and most production agent stacks operate today. But three properties of the agent workload break the assumptions that classical IR systems were optimized around. Classical IR engines amortize ahead-of-time index construction over many queries. Agent memory grows during the query stream, monotonically, at conversation rate . Lucene and PISA both pay full index rebuild cost on update; Mem0 reports 91\ production deployment~. The agent workload demands an index that absorbs writes blocking the adjacent reads on the same agent's reasoning thread. Web search assumes a query distribution slowly drifting over weeks. Agent memory faces an extreme intra-session shift: in the same 800-turn engagement, the agent fires factual lookups in turn 23, multi-step reasoning queries in turn 421, and temporal queries in turn 700. No single fusion maximizes recall across this distribution---we verify this empirically with LongMemEval vs.\ LoCoMo in , where the same hybrid system requires opposite configurations to deliver best quality. Web search has a $ $200\,ms perceptual budget per query and can serve from a shared multi-tenant cluster. An agent step often executes 5--20 retrievals serially against its own memory; the budget per retrieval is therefore $ $10\,ms even on commodity hardware. This is below the regime where Lucene's tuned analyzer is competitive , and well below the regime where dense-encoder pipelines like SPLADE++ can participate . Together these properties rule out the ``small IR'' framing. What the workload demands is a retrieval system that absorbs writes at conversation rate, picks the fusion per query, and serves the agent's reasoning loop at sub-millisecond amortized cost over a corpus that grows into the millions of records during a single session. ~: Strong sparse retrieval but lacks native dense search and agent-aware metadata fusion. ~: Efficient ANN but no BM25, no hybrid fusion, limited metadata filtering. ~: Agent-specific memory management but relies on external retrieval backends without parallelism optimization. closes these gaps by exposing the sparse, dense, and temporal channels through one substrate and choosing among them per query. We document three correctness issues we encountered, fixed, and validated, because they are easy to introduce when re-implementing BM25 and difficult to detect from end-to-end metrics alone. The textbook BM25 saturation function expects the raw integer count $ $: \[ = }{ + k_1\! }. \] A common implementation mistake is to store the relative frequency $/|d|$ in the posting list, reasoning that length normalization will then happen ``naturally.'' This is wrong: the saturation curve and length term in the denominator together are calibrated for raw counts, and pre-normalization normalizes by document length while also flattening the saturation curve. In our initial implementation this regressed nDCG@10 on FiQA from and on SciDocs from ; the regression on NFCorpus was milder because biomedical queries are short and the saturation curve matters less. The original Burges et al.\ formulation uses $ = _i / _2 $ , which is what BEIR/pytrec\_eval report. A linear-gain variant $ = _i r_i/ _2 $ silently differs on graded-relevance datasets while agreeing on binary-relevance datasets . We encountered both formulas in different parts of our codebase; we have standardized on the exponential-gain formula throughout the results in this paper. Our GPU top-$k$ kernel maintains each thread's top-$K_{ }$ in shared memory, with $K_{ }$ being a compile-time constant ($K_{ }{=}32$ for $k 32$; $K_{ }{=}128$ for $k 128$). Each thread writes its valid top-$k$ slots but leaves the remaining $K_{ }{-}k$ slots untouched. Across block re-uses on the same SM, those untouched slots inherit positive scores from a previous block, which can win the round-0 max scan. Learned sparse methods---ColBERT~, SPLADE~, uniCOIL~---produce sparse representations for inverted indexes. RAG~ motivates efficient retrieval in LLM pipelines. Our work targets agent memory with temporal and role-aware fusion not addressed by these systems. The agent-memory literature has so far focused on memory ---what to store, what to consolidate, what to forget---and treated retrieval as a fixed black box. MemGPT~ introduces OS-inspired memory tiers and pages between an LLM context and external storage; the retrieval backend is an off-the-shelf dense or hybrid search. Mem0~ reports 91\ production agent memory through bookkeeping and write-batching optimizations, but again over a generic IR substrate. CoALA~ and Zhang et al.~ taxonomize memory mechanisms without prescribing retrieval architecture. LongMemEval~ and LoCoMo~ provide benchmarks for long-term conversational memory but evaluate retrieval as a single end-to-end metric. 's contribution is orthogonal and complementary: we argue that the retrieval substrate should be agent-aware (temporal partitioning, workload-conditional fusion, correctness-validated GPU acceleration), not just the memory-management layer above it. A production deployment can combine, e.g., Mem0's write-batching policy with 's retrieval substrate and inherit both sets of gains. The Lucene/Anserini~ stack is the de-facto industrial baseline for BM25 and the de-facto reproducibility reference in IR research. PISA~ is a state-of-the-art CPU search engine with aggressive index compression and dynamic pruning. Among GPU-accelerated IR work, Ding et al.~ pioneered GPU query processing; subsequent work~ targets GPU posting list decoding; Griffin~ demonstrates heterogeneous CPU--GPU scheduling . FAISS~ and CAGRA~ provide GPU-optimized ANN. Our contribution is the BM25 path specifically: a CSR-flat inverted index on GPU with a per-(query, term) score kernel plus a per-query top-$k$ kernel, validated for correctness at $k 128$ and benchmarked head-to-head against Pyserini on BEIR. Li and Croft~ formalize temporal language model priors; Campos et al.~ survey temporal IR methods. Our temporal partitioning extends these ideas to agent workloads with extreme recency bias , achieving sub-linear scaling that general temporal IR methods do not target. In practice, agent stacks usually retrieve through a managed vector database---Pinecone, Milvus, Weaviate, Qdrant, or Vespa. These systems are heavily engineered for dense ANN over hosted embeddings; reported $p99$ latency on million-scale corpora is typically 5--50\,ms~, and they bill per read unit on the order of $10^{-7}$\,\$/query. Our work is complementary: we publish an open-source CPU-only BM25 substrate that is several orders of magnitude cheaper per query ($ 10^{-9}$\,\$ on Jetstream2's free , ) and that a managed-VDB deployment can call in parallel with its dense channel. We deliberately benchmark against Pyserini and PISA rather than against managed VDBs because ~the latter conflate retrieval cost with hosted- encoder cost, hiding which layer dominates; and ~Pyserini and PISA are the only systems against which a head-to-head nDCG@10/qrels comparison is reproducible at the quality-and-latency granularity this paper targets. Real deployments will pair this substrate with a managed dense channel; the SPLADE bridge of is the bridge that makes the substrate compatible with a learned encoder service. Agent memory is a different retrieval workload from web search or document ranking: the recency skew is sharper, the corpus grows during the query stream, and the query distribution shifts inside a single session. Treating those properties as design inputs rather than as nuisances is what closes the gap between what generic engines deliver and what an agent's reasoning loop can afford.
