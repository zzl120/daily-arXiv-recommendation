By encoding repository routing knowledge into plain-text catalogs through an adversarial LLM optimization loop, Libra creates a persistent, model-agnostic artifact that continuously improves based on interaction failures. Our findings establish that a system trained entirely on self-generated, synthetic queries can successfully transfer its learned routing capabilities to solve real-world software engineering tasks. Ultimately, Libra demonstrates that optimizing text-based indices via interaction feedback offers a viable and adaptable alternative to traditional retrieval methods. While our results are promising, several limitations present opportunities for future work. First, the LLM optimization loop is computationally intensive; exploring more efficient Healer designs and catalog structures could reduce this overhead. Second, our evaluation assumes static repositories. Because real-world codebases evolve, developing mechanisms for versioning and incremental catalog updates is crucial for practical deployment. Third, the current Prompter primarily generates single-hop queries; synthesizing high-quality, multi-hop queries would better reflect real-world complexity and provide a stronger training signal. Finally, while we focused on software engineering benchmarks using a specific three-agent architecture, extending Libra to other domains and exploring the broader design space of agent interactions remain important directions for future research. Information localization within massive repositories is a cornerstone of agentic LLM systems. While synthetic data-driven optimization has proven successful in training LLMs, little attention has been paid to optimizing the agent's working environment in a data-driven manner. To bridge this gap, we present , a self-evolving framework that introduces mutable ``catalogs'' into the repository. runs an LLM-driven optimization loop where a Prompter generates synthetic queries, a frozen Solver attempts to resolve them by navigating the catalogs, and a Healer rewrites the catalogs in response to the Solver's localization failures. Evaluations across 12 repositories demonstrate that this environmental healing yields continual, logarithmic improvements in code localization accuracy. Furthermore, these environmental improvements transfer zero-shot across different LLMs and problem sets. Although the focus of this paper is to study the general behavior of such a system, we also demonstrate that a minimalist coding agent equipped with -optimized catalogs outperforms state-of-the-art baselines. Repository-scale localization on benchmarks like SWE-bench is dominated by two families: agentic search over the source tree (SWE-agent~, Agentless~, LocAgent~, RepoMem~) and fine-tuned retrievers or models trained on repository data~. Hierarchical or graph-structured retrievers (RAPTOR~, GraphRAG~, HippoRAG~) impose multi-resolution structure on the corpus by clustering or LLM extraction. All of these construct their index, graph, or model weights once and never repair them from query-time failures; Libra keeps the agentic-search interface but turns the index into a state variable updated by Solver failures. LLM-generated queries have a long history as training data for dense retrievers (Doc2Query~, InPars~, Promptagator~, GPL~) and for bootstrapping instruction corpora (Self-Instruct~, Evol-Instruct~). The closest precedent for our information-asymmetry framing is Beat-the-AI~, where annotators exploit oracle access to write hard questions. Libra's Prompter follows this lineage. Many systems improve an agent in place by editing its scratchpad (Reflexion~, Self-Refine~, STaR~), by growing per-agent libraries or memory stores (Voyager~, MemGPT~, A-MEM~), or by routing self-generated trajectories back into model weights (AgentTuning~, Dr.\,Zero~); for software engineering specifically, ExpeRepair~ and SWE-Adept~ attach dual-memory or git-like stores to the issue-resolution agent. In all of these the memory lives inside the agent and is read-write by the agent exclusively; Libra externalizes the memory to a plain-text environment sharable across agents. A growing line of work uses one LLM to optimize a textual artifact that another LLM later consumes: prompts via direct search or via critique-as-gradient (ProTeGi~, TextGrad~); solutions conditioned on score history ; and entire LLM pipelines . Libra's Healer fits this template and applies it to the realm of repository information retrieval. {architecture_v4.png} Code localization within large repositories is a fundamental capability for autonomous coding agents~. Localization accuracy has further been shown to correlate strongly with downstream task resolution rates~, making it one of the key contributors to end-to-end agent performance. More broadly, information retrieval serves as the cornerstone of agentic systems and has consequently been the subject of extensive research~. These agentic systems generally operate on three interacting pillars: the underlying foundation , the , and the itself . To improve an agent's ability to navigate a specific environment, a natural approach is to fine-tune the underlying foundation model using repository-derived data~. While this allows the model to internalize environmental knowledge into its weights using mature optimization techniques, it inherently locks the system to a specific model. Consequently, the agent may lose access to the superior reasoning and semantic capabilities of state-of-the-art commercial LLMs, rendering this approach sub-optimal for practical, evolving applications. Alternatively, researchers have sought to augment the agent's environment by constructing static or reverse indices~ or hierarchical summaries~ and equipping the agent with sophisticated tools to interact with these structures. While these environment-centric approaches are model-agnostic by design, they remain static, lacking a mechanism to adapt dynamically to the agent's specific retrieval challenges or to leverage data---such as human feedback or synthetic interaction traces---to continuously improve the system. A separate line of work leverages interaction trace data, allowing agents to learn from their own failures through adversarial fine-tuning~ or by maintaining persistent memory artifacts within the environment~. Despite these advancements, the field still lacks a , framework that can evolve an agent's information retrieval capabilities for a specific repository. To bridge this gap, we introduce Libra, representing a paradigm shift in repository indexing: moving from static, external embeddings to an actively evolving, plain-text index natively integrated into the repository. Libra constructs a hierarchy of Markdown files, termed , that acts as a navigation map. Instead of relying on predefined heuristics, these catalogs are iteratively optimized through an adversarial LLM-driven training loop. A Prompter agent generates synthetic queries from random code chunks, while a Solver attempts to answer them using only the catalogs for guidance. The Solver's navigation failures provide a rich, targeted training signal for a Healer agent to continuously refine the catalogs, aligning the index structure with how language models actually search for information. We evaluate Libra across the 12 Python repositories of . Our experiments demonstrate the efficacy of this LLM optimization approach: training on synthetic queries yields significant improvements in navigation accuracy that robustly transfer to resolving real-world issues. Furthermore, we show that the fully-trained catalogs act as a persistent, model-agnostic resource, consistently enhancing the localization performance of various downstream models. Compared to state-of-the-art index-based approaches like LocAgent and RepoMem, Libra achieves competitive performance while offering the distinct advantages of interpretability and portability. Both share Libra's overall agentic-localization setup but differ in the the Solver is given: LocAgent exposes a structure-aware repository graph in place of /, while RepoMem layers an additional set of frozen commit-history tools on top of LocAgent. Crucially, neither substrate is rewritten during the run, which is the orthogonal axis Libra adds. LocAgent builds a directed heterogeneous graph over the repository (nodes: directories, files, classes, functions; edges: contain, import, invoke, inherit) and layers a sparse, hierarchical entity index on top of it: ~an entity-ID lookup keyed by fully qualified name, ~an entity-name dictionary keyed by short name (with case-insensitive and splitting fallbacks), ~a BM25 inverted index~ over entity IDs as a fuzzy fallback when / miss, and ~a BM25 inverted index over each entity's code chunks for keywords that never appear in any ID (e.g.\ global variables). The Solver receives three tools in place of Libra's /: --- keyword search through the four-layer cascade above; returns matching entities with their source at one of three detail levels chosen by match count. --- type-aware multi-hop BFS along contain/import/invoke/inherit edges. --- full source retrieval for a chosen node. We use LocAgent's reference repository defaults: each BM25 index is queried at =10, with display further capped at the top 5 hits per query (and at top 3 for non-file/-directory entities in the entity-ID layer). When the entity-ID BM25 layer returns nothing, a small token-set retrieval is consulted before falling through to the content-BM25 layer. RepoMem extends LocAgent with four commit-history tools that expose memory mined offline from the repository's git log: , --- episodic memory, BM25-matched against commit messages from a window of $7{,}000$ prior commits. , --- semantic memory, LLM-generated summaries of the $K{=}200$ most frequently edited files. The memory itself is built once before evaluation and never updated by the Solver; failures cannot feed back into the index. This contrasts with Libra, where the catalog is the artifact under training and is continuously rewritten by the Healer in response to Solver failures.
